# Sprint 1 Database Migration Checklist

Status: Draft pending PM approval

Owner: Database Architecture specialist

Do not sync this document to GitHub until the Product Manager reviews and approves it.

## Purpose

This checklist defines the exact database migration expectations for Sprint 1. Sprint 1 should establish the Supabase foundation for authentication-linked profiles and private reading groups, without implementing later product systems.

Sprint 1 database scope is intentionally small:

- `profiles`
- `groups`
- `group_members`
- helper functions for RLS and group workflows
- initial RLS policies
- initial indexes
- minimal seed/stub guidance

## Source Documents Reviewed

- `12_Database_Architecture.md`
- `18_Sprint_1_Plan.md`

## Sprint 1 Database Outcome

By the end of Sprint 1:

- Authenticated users can create and update their own profile.
- Authenticated users can create a private group.
- Group creators become owner members.
- Authenticated users can join a group through a controlled invite-code flow.
- Active group members can read group details and member lists.
- Non-members cannot read private group data.
- RLS is enabled and active on all Sprint 1 application tables.
- Database constraints prevent invalid core profile and membership states.

## Extensions

Required:

```sql
create extension if not exists pgcrypto;
```

Optional for Sprint 1:

```sql
create extension if not exists citext;
```

Use `citext` only if invite codes or future handles need case-insensitive storage at the database type level. A simpler Sprint 1 option is to normalize invite codes to uppercase in the workflow function.

## Table: `profiles`

Create in Sprint 1.

Required columns:

```sql
id uuid primary key references auth.users(id) on delete cascade,
display_name text not null,
avatar_url text,
bio text,
favorite_genres text[] not null default '{}',
timezone text not null default 'America/New_York',
total_xp integer not null default 0,
current_streak_days integer not null default 0,
longest_streak_days integer not null default 0,
created_at timestamptz not null default now(),
updated_at timestamptz not null default now()
```

Required constraints:

```sql
constraint profiles_display_name_length
  check (char_length(trim(display_name)) between 1 and 80),
constraint profiles_bio_length
  check (bio is null or char_length(bio) <= 500),
constraint profiles_total_xp_nonnegative
  check (total_xp >= 0),
constraint profiles_current_streak_nonnegative
  check (current_streak_days >= 0),
constraint profiles_longest_streak_nonnegative
  check (longest_streak_days >= 0)
```

Sprint 1 notes:

- Keep XP and streak columns as placeholders with defaults.
- Do not implement XP or streak logic in Sprint 1.
- Do not add public usernames or follower fields.

## Table: `groups`

Create in Sprint 1.

Required columns:

```sql
id uuid primary key default gen_random_uuid(),
name text not null,
description text,
owner_id uuid not null references profiles(id) on delete restrict,
invite_code text unique,
is_archived boolean not null default false,
created_at timestamptz not null default now(),
updated_at timestamptz not null default now()
```

Required constraints:

```sql
constraint groups_name_length
  check (char_length(trim(name)) between 1 and 100),
constraint groups_description_length
  check (description is null or char_length(description) <= 1000),
constraint groups_invite_code_format
  check (invite_code is null or invite_code ~ '^[A-Z0-9]{6,12}$')
```

Sprint 1 notes:

- `owner_id` should point to the creator's profile.
- `invite_code` should be generated by trusted database or server logic, not typed directly by arbitrary client writes.
- Keep `is_archived`; do not build archive UI unless Sprint 1 scope explicitly includes it.

## Table: `group_members`

Create in Sprint 1.

Required columns:

```sql
id uuid primary key default gen_random_uuid(),
group_id uuid not null references groups(id) on delete cascade,
user_id uuid not null references profiles(id) on delete cascade,
role text not null default 'member',
status text not null default 'active',
joined_at timestamptz not null default now(),
last_seen_at timestamptz
```

Required constraints:

```sql
constraint group_members_unique_group_user
  unique (group_id, user_id),
constraint group_members_valid_role
  check (role in ('owner', 'admin', 'member')),
constraint group_members_valid_status
  check (status in ('invited', 'active', 'left', 'removed')),
constraint group_members_last_seen_after_join
  check (last_seen_at is null or last_seen_at >= joined_at)
```

Recommended ownership invariant:

- The `create_group` workflow must insert exactly one initial `owner` membership for the group creator.
- Do not try to enforce "exactly one owner" with a complex trigger in Sprint 1 unless engineering explicitly wants that guardrail.
- Add a partial unique index for active owner membership if desired:

```sql
create unique index group_members_one_active_owner_per_group
  on group_members (group_id)
  where role = 'owner' and status = 'active';
```

Sprint 1 notes:

- Former members should remain as `left` or `removed` rather than being hard-deleted by default.
- Member role editing can be deferred; only owner creation and member join are required.

## Updated-At Trigger

Sprint 1 should add a reusable `updated_at` trigger for tables that have `updated_at`.

Expected function:

```sql
create or replace function public.set_updated_at()
returns trigger
language plpgsql
as $$
begin
  new.updated_at = now();
  return new;
end;
$$;
```

Attach to:

- `profiles`
- `groups`

Do not attach to `group_members` unless an `updated_at` column is added.

## RLS Helper Functions

Create helper functions before policies.

All helper functions should:

- Live in `public`.
- Be `security definer`.
- Set `search_path = public`.
- Use `auth.uid()` for the current user.
- Avoid returning more information than needed.

### `owns_profile(user_id uuid)`

Expected behavior:

- Returns true when `auth.uid() = user_id`.

Draft SQL:

```sql
create or replace function public.owns_profile(user_id uuid)
returns boolean
language sql
stable
security definer
set search_path = public
as $$
  select auth.uid() = user_id;
$$;
```

### `is_group_member(group_id uuid)`

Expected behavior:

- Returns true when the current user has an active membership in the group.

Draft SQL:

```sql
create or replace function public.is_group_member(group_id uuid)
returns boolean
language sql
stable
security definer
set search_path = public
as $$
  select exists (
    select 1
    from public.group_members gm
    where gm.group_id = is_group_member.group_id
      and gm.user_id = auth.uid()
      and gm.status = 'active'
  );
$$;
```

### `is_group_admin(group_id uuid)`

Expected behavior:

- Returns true when the current user has active `owner` or `admin` membership.

Draft SQL:

```sql
create or replace function public.is_group_admin(group_id uuid)
returns boolean
language sql
stable
security definer
set search_path = public
as $$
  select exists (
    select 1
    from public.group_members gm
    where gm.group_id = is_group_admin.group_id
      and gm.user_id = auth.uid()
      and gm.status = 'active'
      and gm.role in ('owner', 'admin')
  );
$$;
```

### `is_group_owner(group_id uuid)`

Recommended for Sprint 1 even if not required by the database architecture doc.

Expected behavior:

- Returns true when the current user has active `owner` membership.

Draft SQL:

```sql
create or replace function public.is_group_owner(group_id uuid)
returns boolean
language sql
stable
security definer
set search_path = public
as $$
  select exists (
    select 1
    from public.group_members gm
    where gm.group_id = is_group_owner.group_id
      and gm.user_id = auth.uid()
      and gm.status = 'active'
      and gm.role = 'owner'
  );
$$;
```

## Workflow Functions

Sprint 1 should prefer trusted database functions or server-side actions for multi-table group workflows.

### `create_group(name, description)`

Required in Sprint 1 if group creation is implemented.

Expected behavior:

- Requires an authenticated user.
- Requires the authenticated user to already have a profile.
- Inserts a `groups` row.
- Generates a normalized invite code.
- Inserts an active owner row into `group_members`.
- Returns the created group id and invite code.
- Runs atomically.

Validation:

- Group name is trimmed.
- Group name length is 1-100 characters.
- Description is optional and length-limited.
- Invite code collision is retried or handled safely.

Security:

- Function should not accept `owner_id` from the client.
- Function should use `auth.uid()` as the owner.

### `join_group_by_invite(invite_code)`

Required in Sprint 1 if join-by-code is implemented.

Expected behavior:

- Requires an authenticated user.
- Requires the authenticated user to already have a profile.
- Normalizes invite code.
- Finds a non-archived group.
- Inserts active member row if no membership exists.
- Reactivates `left` membership if PM approves that behavior.
- Returns group id after successful join.
- Returns a generic error for invalid code.

Security:

- Must not reveal private group information before successful membership creation.
- Must not allow joining archived groups.
- Must not allow `removed` users to rejoin unless PM explicitly approves.

## RLS Enablement

Enable RLS on every Sprint 1 application table:

```sql
alter table public.profiles enable row level security;
alter table public.groups enable row level security;
alter table public.group_members enable row level security;
```

Recommended:

```sql
alter table public.profiles force row level security;
alter table public.groups force row level security;
alter table public.group_members force row level security;
```

Use `force row level security` only after confirming it does not break trusted workflow functions.

## RLS Policies: `profiles`

Required read policy:

- Users can read their own profile.
- Users can read profiles for active members of groups they belong to.

Expected policy shape:

```sql
create policy "profiles_select_self_or_group_members"
on public.profiles
for select
to authenticated
using (
  public.owns_profile(id)
  or exists (
    select 1
    from public.group_members viewer_membership
    join public.group_members target_membership
      on target_membership.group_id = viewer_membership.group_id
    where viewer_membership.user_id = auth.uid()
      and viewer_membership.status = 'active'
      and target_membership.user_id = profiles.id
      and target_membership.status = 'active'
  )
);
```

Required insert policy:

- Users can insert only their own profile.

Expected policy shape:

```sql
create policy "profiles_insert_self"
on public.profiles
for insert
to authenticated
with check (public.owns_profile(id));
```

Required update policy:

- Users can update only their own profile.

Expected policy shape:

```sql
create policy "profiles_update_self"
on public.profiles
for update
to authenticated
using (public.owns_profile(id))
with check (public.owns_profile(id));
```

Delete policy:

- Do not add a client delete policy in Sprint 1.
- Account deletion should be handled separately through approved auth/account workflows.

## RLS Policies: `groups`

Required read policy:

- Active members can read their groups.

Expected policy shape:

```sql
create policy "groups_select_active_members"
on public.groups
for select
to authenticated
using (public.is_group_member(id));
```

Insert policy:

- If using `create_group` RPC with elevated or definer behavior, direct client insert can be omitted.
- If direct insert is allowed, constrain it to `owner_id = auth.uid()` and pair it with application/server logic that creates the membership immediately.

Recommended Sprint 1 policy:

```sql
create policy "groups_insert_owner_self"
on public.groups
for insert
to authenticated
with check (owner_id = auth.uid());
```

Update policy:

- Owners/admins can update basic group details.

Expected policy shape:

```sql
create policy "groups_update_admins"
on public.groups
for update
to authenticated
using (public.is_group_admin(id))
with check (public.is_group_admin(id));
```

Delete policy:

- Do not add a client delete policy in Sprint 1.
- Use `is_archived` for soft archive later.

## RLS Policies: `group_members`

Required read policy:

- Active members can read membership rows for their groups.

Expected policy shape:

```sql
create policy "group_members_select_group_members"
on public.group_members
for select
to authenticated
using (public.is_group_member(group_id));
```

Insert policy:

- Prefer joining through `join_group_by_invite`.
- Direct inserts by ordinary clients should be avoided unless tightly constrained.

Recommended Sprint 1 policy:

```sql
create policy "group_members_insert_self_member"
on public.group_members
for insert
to authenticated
with check (
  user_id = auth.uid()
  and role = 'member'
  and status = 'active'
);
```

Important:

- If this direct insert policy creates any path to join groups without invite validation, omit it and require the RPC.
- Owner membership should be created only by `create_group`.

Update policy:

- Users may mark their own membership as `left`.
- Admin role changes can be deferred.

Recommended conservative Sprint 1 option:

```sql
create policy "group_members_update_self_leave"
on public.group_members
for update
to authenticated
using (user_id = auth.uid())
with check (
  user_id = auth.uid()
  and status in ('active', 'left')
  and role = group_members.role
);
```

If role comparison in `with check` is not practical in the implementation, defer self-leave updates and handle leaving groups through a trusted function.

Delete policy:

- Do not add a client delete policy in Sprint 1.
- Membership records should be retained for audit and history.

## Indexes

Required Sprint 1 indexes:

```sql
create index profiles_display_name_idx
  on public.profiles (display_name);

create index groups_owner_id_idx
  on public.groups (owner_id);

create unique index groups_invite_code_unique_idx
  on public.groups (invite_code)
  where invite_code is not null;

create index group_members_user_status_idx
  on public.group_members (user_id, status);

create index group_members_group_status_idx
  on public.group_members (group_id, status);
```

Recommended Sprint 1 index:

```sql
create unique index group_members_one_active_owner_per_group
  on public.group_members (group_id)
  where role = 'owner' and status = 'active';
```

Avoid in Sprint 1:

- Book search indexes.
- Feed indexes.
- Scoring indexes.
- Notification indexes.
- Review indexes.

## Seed And Stub Guidance

Do seed:

- Optional local-development test achievements only if the table exists, but Sprint 1 should not create achievement tables by default.
- Optional local-only sample groups or profiles only in a clearly separated development seed file.

Do not seed:

- Production users.
- Production groups.
- Fake private group memberships in shared environments.
- XP events.
- Momentum scores.
- Notifications.
- Reviews.
- Book of the Month data.

Recommended Sprint 1 seed posture:

- Keep production migrations schema-only.
- Put local development seed data in a separate file that is never run automatically in production.
- Use Supabase Auth test users through local tooling rather than inserting directly into `auth.users` in normal migrations.

## Recommended RLS Test Cases

Create automated or manual verification for these cases:

- User A can read and update User A profile.
- User A cannot update User B profile.
- User A cannot read User B profile before sharing a group.
- User A can create a group and becomes owner.
- User B cannot read User A's group before joining.
- User B can join with a valid invite code.
- User B can read the group after joining.
- User A and User B can read each other's profile after sharing an active group.
- User C with no membership cannot read the group or member list.
- Invalid invite code returns a generic error.
- Archived group cannot be joined.
